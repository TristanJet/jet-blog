### The problem
Compartmentalization comes with it's benefits. Distinguishing between a device for communication and one for entertainment.

**Abstraction reduces required knowledge while limiting behaviour and restricting access**

As a computer is programmed to *do* more things, it's repertoire of things it *can do* decreases.

Programming a computer can be thought of like carving a statue from a block of stone. Increasingly the number of things the program will *never* do increases, as the the program becomes more *useful*. The programmer narrows down the possible computations for the machine into a useful subset.

Turing describes the **universal Turing machine** as a machine that accepts **descriptions of non-universal machines**, which the universal machine then outputs the same output as the which the non-universal machine would output i.e. *simulates*.

The process of limiting behaviour transforms infinite possibility into useful reality. Something that *can* be anything *is* nothing. A blank sheet of paper.

There is an enumerable infinity of possible programs for a Universal Turing Machine, most of which, are *not* productive. 

Some specific behaviours are actually harmful, as they break programs completely, so much so that programmers go to great lengths to guarantee that these so-called *unsafe* behaviours should *never* occur. Fortunately, unsafe behaviours are well-known and understood.

The history of software is a history of programmers creating abstractions for, and enforcing rules on others in order to access computers more simply.

Some examples:

Compilers enforcing strict type systems
Kernels protecting user-space programmers from ever accessing kernel-space memory. 
Wrapping things in functions.
Using asserts.
